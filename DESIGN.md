## Problem Statement

Korean amateur 9인제 (9-man) volleyball clubs are run by 1-3 unpaid managers — typically a captain (캡틴) and a secretary (총무). Their job is split across four disconnected systems:

- **Daum cafes** (e.g., 'Let's go 생활체육배구') for tournament announcements
- **Naver Band** for club announcements, photos, attendance polls
- **KakaoTalk** for day-to-day chat
- **KakaoBank 모임통장** for collecting and reconciling 회비 (dues)

The captain manually checks the Daum cafe for new tournaments, drafts pre-formatted Band posts for attendance, and the secretary manually reconciles bank statements against the member roster every cycle. None of this is automated, and no existing tool unifies it. The user is a regular player in such a club and is building this as a CV/portfolio piece for a job hunt — the goal is a shipped native Android app with real captain users that demonstrates product judgment, technical depth, and follow-through. iOS port comes later as v2.

## What Makes This Cool (v3 — reframed by CEO review 2026-04-08)

**The reframed thesis:** Libero is **the operations layer for amateur club captains**. Korean 9인제 captains run their clubs across 4 disconnected systems (Daum cafe, Naver Band, KakaoTalk, KakaoBank) and the captain is doing invisible labor that nothing captures. Libero is the layer that doesn't replace any of those systems but connects them and saves the captain time on the work nobody sees. Multi-manager (captain + secretary) is a *capability*, not the *headline*.

**Module 1 leads on demo material.** A push notification arrives within hours of a tournament being posted in the cafe — the captain taps and lands instantly on the tournament detail screen with all data populated from the FCM payload (no spinner). The killer demo is the visceral "you got pushed → you're already in the detail screen." This is the screen you record for the CV demo video.

> **Reality check (per CEO review interview question A3):** if your captain has Daum cafe app push enabled, Libero is the *second*-fastest source not the first. The actual differentiator becomes the *filter and structured-data* play (regions, divisions, deadlines, dedup, parser confidence) — not raw speed. Verify in the captain interview before W1.

**Module 2 is the CV depth play.** The product judgment story: Vision OCR was the obvious answer — Libero routes around the entire fragile-OCR problem by using KakaoBank's `거래내역 다운로드` Excel export and fuzzy-matching names against the roster on-device. 100% data accuracy where Vision OCR would have been ~70-95% on a good day. **This is the design decision senior engineers respect**, the one you tell in interviews when asked "what was your hardest design call." The flow is the *real* multi-manager moment: secretary uploads from her phone (where the bank login lives), captain sees the result update live via filtered Realtime.

> **Reality check (per CEO review interview question B2):** does the secretary actually want to switch from her current Excel reconciliation, or is the status quo "5 minutes eyeballing the spreadsheet" already fine? Verify in the captain interview before W1.

**Module 3 is the breadth play and the Pro tier hook.** Auto-posts the weekly practice poll to Naver Band on a schedule. Captain configures once, the app runs the recurring post forever. The only feature that saves time *every single week* — load-bearing for the future Pro tier value prop. CEO review upgraded M3 from W9 cut-valve to PROTECTED in MVP because cutting it would leave the reframed thesis hollow at launch.

**Multi-manager is a capability that surfaces in Module 2.** Captain and secretary both sign in on their own phones, see the same data, edit the same records via filtered Realtime. Conflict handling is silent last-write-wins. v1 has zero role-based UI branching — every user inside a club sees identical screens with identical permissions, regardless of label. Role differentiation is v2 work (TODOS.md).

**Members never install.** Zero adoption friction, zero dual-tool fatigue. The app is invisible to the people the club serves. The captain processes member data on their behalf — and per PIPA, the captain is the data controller and must notify members. CEO review added the W4 captain-attestation checkbox to make this notice mechanism real, not bureaucratic.

**Built for the platform Korean amateur captains actually use.** Android-first, not iOS-first. Your captain owns a Galaxy S-something, not an iPhone. Match the user, not the developer aesthetic.

## Constraints

- Single developer, **~13 weeks committed** for MVP (see Effort section). Includes the Google Play 14-day closed-testing requirement (locked as applicable per user's homework — new personal Play Console account).
- Budget: **$25 one-time** (Google Play Console) + ~$1-3/mo (Anthropic Haiku) + $0 recurring otherwise. **No domain purchased** (Decision Reversal 4+5).
- Korean Google Play Store first, US Play Store later, **iOS App Store as v2 (post-MVP)**
- Native Android only (not web, not cross-platform — match the platform Korean amateur captains actually use; iOS port deferred to v2)
- Pre-build: 2 captain interviews mandatory before any code (own captain + own secretary, with the new CEO-review-added questions in `notes/captain-interview-script.md`)
- Personal Daum account used for scraping (user-accepted risk — see Reviewer Concerns)
- **Scraper polling: 3 hours, locked.** This is the chosen point inside the 2-4 hour stealth envelope (see Reviewer Concerns). Tune after first month of real use based on `time_from_post_to_notification` metric.
- **Min Android version: API 26 (Android 8.0)** — covers ~98% of active Korean Android devices in 2026, allows modern Compose features without legacy workarounds. Re-evaluate if your captain runs anything older.
- **Google Play Console: confirmed NEW personal developer account.** Google's Nov 2023 policy requires new personal accounts to run a closed test with **12+ testers for 14 continuous days** before production publishing is unlocked. This adds 2 weeks to the launch phase (W11-W13). The 14-day clock is calendar days, not work days.
- **Invite mechanism: 6-character alphanumeric codes** (Decision Reversal 4). NOT App Links, NOT magic links, NOT domain-based. Codes have 7-day TTL and can be rotated. KakaoTalk Sharesheet handles the share UX without any KakaoTalk SDK.

## Premises

1. **Korean 9인제 captains have a real, unmet workflow problem** — verified by user being in a 9인제 club with personal access to the captain
2. **Multi-manager (captain + secretary) is the right scope** — verified, single-user breaks how clubs operate
3. **The 'Let's go 생활체육배구' Daum cafe is technically scrapeable** — verified at the access level (등업 required, user has account, no posting requirement). **Not yet verified at the technical level — see Week-1 Prototype Gate.**
4. **Naver Band API is alive in 2026 and self-serve** — verified by user. **Posting scope still needs verification** (write endpoints historically require separate approval).
5. **KakaoBank 거래내역 download exists and exports a structured Excel file** — confirmed by user. **Exact column structure unverified** — gating Module 2 estimate.
6. **Tournament posts require LLM extraction** — agreed, free-form Korean prose can't be regexed reliably
7. **CV story is general portfolio (not Korean-specific or US-specific)** — bilingual is post-MVP work
8. **Korean amateur 9인제 captains (age 30-50) overwhelmingly use Android, not iOS.** Verified by Korean smartphone market data: ~70%+ Android share overall, even higher in the 30-50 cohort. iOS in Korea is concentrated in teens/20s as a fashion item. Building iOS-first means shipping to a minority of the target user base. Added 2026-04-08 PM as the basis for the Android-first pivot.

## Approaches Considered

### Approach A: Captain-only local-first
Single-user app, local-only, no auth, no backend except for the scraper.
- **Pros:** simplest to build, no auth complexity, no cloud cost beyond scraper
- **Cons:** secretary can't use it, breaks how real clubs operate
- **Effort:** ~6-7 weeks
- **Verdict:** Rejected — user confirmed multi-manager is the real need.

### Approach B: Multi-manager Supabase + 3-module MVP, native Android **← RECOMMENDED**
Captain + secretary both sign in via Sign in with Google (Kakao Login as alternate), shared club data via Supabase, native Kotlin + Jetpack Compose Android app.
- **Pros:** matches real workflow AND matches the platform Korean amateur captains actually use; supports a strong CV story (multi-tenant case study + product judgment + Korean cultural depth + native Android depth)
- **Cons:** Supabase + RLS + invite flow + FCM adds real auth/infra surface; iOS users (a minority of the target demographic) wait for v2
- **Effort:** ~9-10.5 weeks MVP (honest), +1-2 weeks for v1.1 lineup planner, +6-8 weeks for v2 iOS port (separate)
- **Reuses:** Supabase auth, Sign in with Google, fastexcel-reader for Excel parsing, Apache Commons Text for fuzzy match, Claude Haiku for tournament parsing, Naver Band Open API, Supabase Edge Functions for FCM dispatch

### Approach C: Kotlin Multiplatform (KMP) — share business logic across Android + iOS (lateral, considered 2026-04-08 PM)
Write the data layer, networking, models, and module logic once in Kotlin. Compose UI for Android now, SwiftUI UI for iOS later, sharing 60-70% of the code via KMP.
- **Pros:** When you do ship iOS as v2, you reuse most of the work. Stronger long-term CV story ("shipped Android first, ported to iOS reusing 70% of business logic via KMP"). Great if you have a clear iOS roadmap.
- **Cons:** KMP setup adds 1-2 days of friction now. Swift interop has rough edges, especially around Coroutines + callbacks. Splits your attention across two language ecosystems.
- **Verdict:** Rejected for MVP. Right tool for v2 if iOS becomes a confirmed roadmap item, not a "maybe later."

### Approach D: Flutter — single Dart codebase, both stores (lateral, considered 2026-04-08 PM)
- **Pros:** Ships to Android and iOS simultaneously, fastest path to "any platform."
- **Cons:** Not native. Korean tech employers (Naver, Kakao, Toss, Coupang) mostly run native shops. The CV story gets blurry — "I built a Flutter app" reads less senior than "I built a native Android app." Also gives up the platform-depth angle entirely.
- **Verdict:** Rejected — the CV story rests on platform depth, not platform breadth.

## Recommended Approach: B — Multi-manager Supabase, native Android, MVP

### Week-1 Prototype Gate (CRITICAL)

Before any other build work, spend the first week proving the **two load-bearing assumptions** that the entire design rests on. If either fails, the whole project needs rethinking before any Android app work begins.

**Gate 1A — Daum scraper feasibility.** Cloudflare Workers have no headless browser. If Daum requires JS execution, dynamic CSRF tokens, or anti-bot challenges that need a real browser, a fetch-only Worker scraper breaks.

1. Locally script a `fetch` call against the cafe with the user's logged-in Daum cookies. Confirm posts are returned in HTML.
2. If fetch-only works: deploy as Cloudflare Worker, confirm cookies survive the env-var serialization.
3. If fetch-only fails: pivot to **Plan B** — host the scraper on a small Fly.io or Railway instance running Playwright headless Chrome. Adds ~$5/mo and +2-3 days of setup, but unblocks the project.
4. Document cookie freshness window: how long does a Daum session last before re-login is needed?

**Gate 1B — fastexcel-reader smoke test on a real KakaoBank export.** Module 2's entire architecture rests on parsing the bank's `.xlsx` correctly on-device. If fastexcel-reader chokes on Korean column headers, merged cells, or summary rows, Module 2 needs a different parser.

1. Get a real KakaoBank 거래내역 다운로드 export from your secretary (the homework you're doing anyway).
2. In a JVM scratchpad project (no Android, just `gradle init` and a `main()`), feed the file to fastexcel-reader.
3. Confirm: column headers extract correctly, all transaction rows are read, name field is in the expected column, no UTF-8 corruption, no merged-cell crashes.
4. If fastexcel-reader fails on the real file: try **POI XSSF** in the same scratchpad (POI is fine on JVM/Android, just adds ~5-10MB to the APK if used on-device — acceptable hit if it's the only option). If neither works, the design needs a parser rethink.

**Gate decision:** if either gate fails and no fallback works in week 1, the killer feature(s) are not feasible and the design needs rethinking before any Android app work begins.

### System Architecture

```
[Android App (Jetpack Compose)]
    ↕ Supabase Kotlin SDK (Auth + Postgres + Realtime)
    ↕ Firebase Cloud Messaging (FCM)

[Supabase] ← source of truth
    - Postgres (clubs, members, payments, tournaments, prefs, aliases)
    - Auth (Sign in with Google, with Kakao Login as alt provider)
    - Edge Functions (push dispatch, Band API calls, invite-link minting)
    - RLS policies enforce per-club data isolation

[Cloudflare Worker — scraper] (or Fly.io + Playwright if fetch-only fails)
    - Cron: every 3 hours (locked, within 2-4h stealth envelope)
    - Logs into Daum with cookies from env var
    - Fetches new posts from 'Let's go 생활체육배구'
    - Calls Anthropic Haiku to extract tournament fields
    - Writes to Supabase tournaments table
    - Triggers Supabase Edge Function to fan out FCM push
```

**RLS policies (sketch):** every table with `club_id` enforces `club_id IN (SELECT club_id FROM club_managers WHERE user_id = auth.uid())`. Tables without `club_id` (`users`, `push_tokens`) enforce `user_id = auth.uid()`. `tournaments` is global (not club-scoped) but read-only to clients; only the scraper service-role key writes.

**Invite flow (end-to-end):**
1. Existing manager taps "Invite secretary" in club settings
2. Android app calls Supabase Edge Function `mint_invite` → Edge Function inserts row into `club_invites` with new UUID token, `expires_at = now() + 7 days`, `used_at = null`, returns the token
3. App constructs an invite URL: `https://<your-domain>/invite/<token>` and launches an Android Sharesheet via `Intent.ACTION_SEND` with `type = "text/plain"` and the URL as `EXTRA_TEXT`. KakaoTalk appears in the picker because it's installed on every Korean phone and registers an `ACTION_SEND text/plain` intent filter. No KakaoTalk SDK needed.
4. New manager taps the URL on their Android phone → Android **App Links** (verified deep link via `assetlinks.json` hosted at `https://<your-domain>/.well-known/assetlinks.json`) opens the app at the invite-redemption screen
5. New manager signs in with Google (Sign in with Google is the only auth provider at MVP; Kakao Login is deferred to v1.1) if not already signed in
6. App calls Edge Function `redeem_invite(token)` → Edge Function validates: row exists, `used_at IS NULL`, `expires_at > now()` → inserts `(club_id, user_id, joined_at)` into `club_managers` → updates `club_invites.used_at = now()` → returns club info

**App Links setup notes (gotchas):**
- The domain must serve `assetlinks.json` over HTTPS (not HTTP) **before first install** or Android Verified Links will not register the app as the default handler. Use Cloudflare Pages or a static site host, not the Cloudflare Worker scraper.
- The `assetlinks.json` SHA-256 fingerprints must include both the **Play Store upload key** AND the **Play App Signing key** (Google re-signs all apps with their own key on Play Store distribution). Get the Play App Signing key SHA-256 from Play Console → Setup → App signing.
- Test with `adb shell pm verify-app-links --re-verify <package>` and `adb shell pm get-app-links <package>` to confirm verification status.

**Sign in with Google setup notes (gotchas):**
- Supabase's Google provider needs an **OAuth 2.0 Web client ID** configured in Google Cloud Console (used as the `client_id` Supabase passes to Google's OAuth endpoint). It also needs an **Android client ID** with the SHA-1 of the app's signing key, registered in the same Google Cloud project. **Both are required.** Most first-time errors here come from forgetting the Android client ID.
- Use the Play App Signing key SHA-1, NOT the upload key SHA-1, for the production Android client ID. Use the debug keystore SHA-1 for local development. Register both.

8. **Expiry handling:** if `expires_at < now()`, redeem fails with "이 초대 링크는 만료되었습니다" → user asks the existing manager for a fresh link
9. **Already-used handling:** if `used_at IS NOT NULL`, redeem fails with "이 초대 링크는 이미 사용되었습니다"

**Realtime sync (multi-manager edits):** Supabase Realtime subscriptions on `member_payments`, `payment_cycles`, `members`, and `tournament_preferences` keyed by `club_id`. When manager A edits a row, manager B's app receives the update within 1-2 seconds via WebSocket and updates Room via the Kotlin SDK's Flow API. **Conflict handling is silent last-write-wins** at row granularity. The loser does not see a toast or warning, just the new value via the realtime subscription. Per-field change tracking ("updated by [name]") was considered and rejected as not worth the complexity at MVP scale.

**Offline writes (Room as cache):** Room is the local cache and the source of UI state via Compose `collectAsState`. Writes go to Supabase first; on success, Room is updated via the Realtime subscription. On network failure, writes are queued in a `pending_writes` Room table and retried on reconnect. Queue order is preserved per club. This is sufficient for two managers in a single club; full CRDT-style offline merge is deferred to post-MVP if real users complain.

### Push Dispatch Path

- **Scraper writes** new tournament rows to Supabase via service-role key
- **Postgres trigger** on `tournaments` insert calls a **Supabase Edge Function** `dispatch_tournament_match`
- Edge Function: query all `tournament_preferences` where regions/division/date match → for each matched club → query `club_managers.user_id` → query `push_tokens` → call **FCM HTTP v1 API** (`https://fcm.googleapis.com/v1/projects/{project}/messages:send`) with the device token and notification payload
- **FCM auth:** Edge Function uses a Firebase Admin SDK service account JSON (download from Firebase Console → Project Settings → Service Accounts; store the entire JSON as a Supabase secret named `FIREBASE_SERVICE_ACCOUNT_JSON`). Mint a short-lived OAuth bearer token by signing a JWT with the service account private key (RSA-SHA256), then exchange it at `https://oauth2.googleapis.com/token`. Use the [`jose`](https://deno.land/x/jose) Deno library for JWT signing (RSA-SHA256 in 5 lines, vs ~40 lines of Web Crypto glue). Cache the resulting access token for its 1-hour TTL inside the Edge Function via a module-level variable. Required service account role: **Firebase Admin SDK Administrator Service Agent** (auto-assigned when downloading the service account JSON from Firebase Console — do not assign manually).
- Single dispatch path. Cloudflare Worker is for scraping only; Supabase Edge Functions handle all push and Band API calls.

#### Module 2: 모임통장 결제 추적기 — Payment Reconciliation (Excel-based)

  1. Secretary requests 거래내역 다운로드 from KakaoBank → email arrives with `.xlsx` (verified in 2026 — KakaoBank exports xlsx, not legacy xls)

Below 0.85 = prompt. **Library:** hand-rolled normalized Levenshtein in ~30 lines of Kotlin (preferred — no APK bloat). Apache Commons Text `LevenshteinDistance` is a fine fallback if you'd rather pull a dependency, but it adds ~1.5MB to the APK for one function. Code sketch:
     ```kotlin
     fun similarity(a: String, b: String): Double {
       val distance = levenshtein(a, b)  // standard 30-line DP
       val maxLen = maxOf(a.length, b.length)
       return if (maxLen == 0) 1.0 else 1.0 - distance.toDouble() / maxLen
     }
     ```

## Privacy / PIPA Considerations (Korea)

A Korean app handling personal data (member names, KakaoTalk handles, payment records, bank transaction data) is subject to PIPA (개인정보 보호법). Required:

- **Privacy policy** in Korean, linked from app + Play Store listing. Specifies what data is collected, why, where it's stored, and retention.
- **Minimum data collection**: member rosters and payment data are stored in Supabase (encrypted at rest). Bank export Excel files are parsed locally; only the parsed match results (name, amount, paid status) are synced — never the raw Excel.
- **Daum cookies** are stored as Cloudflare Worker secrets (encrypted at rest). They're scraping infrastructure, not user data, but should be rotated per the cookie freshness window.
- **No third-party analytics** at MVP. Add later only with explicit consent.
- **Data deletion**: managers can delete the club entirely from settings → cascade deletes all related rows.
- **Google Play Data Safety form**: declare collected data types — name (linked to user), contacts (member roster), financial info (payment records). Declare encryption in transit (Supabase HTTPS) and at rest (Postgres). Declare data deletion path. Korean Play Store reviewers scrutinize financial-data apps; the privacy policy URL must resolve to a real Korean-language page on submission day.
- **Android-specific permissions:** `INTERNET` (always), `POST_NOTIFICATIONS` (Android 13+, request at runtime on first launch). **Do NOT request `READ_MEDIA_IMAGES` at MVP** — the lineup template thumbnails come with Module 3, which is deferred to v1.1. Use SAF (`ACTION_OPEN_DOCUMENT`) for the KakaoBank Excel upload, which does not require any storage permission. Smaller permission set = less Play Store data safety friction = faster review.
